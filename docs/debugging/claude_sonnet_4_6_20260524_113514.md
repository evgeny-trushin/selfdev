# Debuggable Multi-Agent AI Coding: Techniques, UX Patterns, and Architecture for Inspectable, Trustable Code Changes

## TL;DR
- **The bottleneck has shifted from generation to inspection**: agents produce far more code than humans can read line-by-line — Veracode's Spring 2026 GenAI Code Security Update reports "security pass rates have remained stubbornly flat at approximately 55%…near-perfect syntax correctness rates exceeding 95%" — so the 2026 discipline is *making agent work legible* through provenance ledgers (Git AI, Mesa Agent Blame), event-sourced replay (OpenHands V1, LangGraph, AGDebugger), semantic diffs (RefactoringMiner, diffsitter), and OpenTelemetry-standard agent traces.
- **A debuggable multi-agent coding platform should rest on six primitives**: per-agent isolation (git worktrees/sandboxes), an append-only event log per agent with deterministic replay, line- and AST-level provenance carried in git notes, semantic + visual diffs grouped by intent, Playwright trace + visual-regression evidence on every change, and a PR review surface that exposes risk, confidence, and the prompt/context the agent saw.
- **Practical roadmap**: in 30 days ship worktree-isolated agents + git-notes provenance + OpenTelemetry GenAI tracing; in 90 days add semantic diff grouping, Playwright traces, risk/confidence cards; in 6 months build replayable agent timelines (AGDebugger-style) + a conflict-resolution view; in 12 months ship a full deployment-readiness dashboard with accessibility/security/PII diffs and a critic-model gate.

## Key Findings

1. **Multi-agent coding fails in characteristic, predictable ways.** The AGDebugger paper (Epperson et al., CHI 2025) identified three core challenges in formative interviews with agent developers: "difficulty reviewing long agent conversations to localize errors, lack of support in current tools for interactive debugging, and the need for tool support to iterate on agent configuration." On long-horizon work, the SWE-EVO benchmark paper (arXiv 2512.18470, December 2025) reports that "GPT-5 with OpenHands achieves only 21 percent resolution rate on [SWE-EVO], compared to 65 percent on the single-issue SWE-Bench Verified." That gap is what production debuggability tooling exists to close.

2. **The industry has converged on OpenTelemetry GenAI semantic conventions** (`gen_ai.agent.id`, `gen_ai.agent.name`, `gen_ai.agent.version`, `gen_ai.conversation.id`, `gen_ai.tool.type`, `invoke_agent`/`create_agent` spans). Datadog, Honeycomb, and New Relic already ingest these, and Langfuse, Arize Phoenix, LangSmith, Helicone, and Braintrust support them. A vendor-neutral observability spine for multi-agent coding is possible today.

3. **Event-sourced agent state is the production default.** OpenHands V1 (arXiv 2511.03690) defines "an event-sourced state model with deterministic replay, an immutable configuration for agents, and a typed tool system with MCP integration." LangGraph ships a canonical `StateSnapshot` JSON `{v, ts, id, channel_values, channel_versions, versions_seen, parent_checkpoint_id}`. AGDebugger checkpoints state before every message so users can "reset to earlier points in a conversation and edit agent messages."

4. **Parallel agents now run in git worktrees as standard practice.** Claude Code supports `--worktree` / `isolation: worktree` for subagents; Cursor 3's Agents Window orchestrates worktrees natively (`/worktree`, `/best-of-n` for parallel-model runs with aggregated diff viewer, `/multitask` async subagents from April 2026). Worktrees solve file-edit conflicts at the filesystem level; semantic conflicts still need diff/test tooling on top.

5. **Line-level AI provenance is now a real product category.** Git AI and Mesa's Agent Blame both extend `git blame` with AI attribution by writing into git notes (`refs/notes/ai`), carrying (agent_id, model, session, prompt) metadata forward. Git AI's blame command is wire-compatible with `git blame`: "git-ai blame path/to/file is a drop-in replacement for git blame that includes the AI authorship attribution for each line." Git AI explicitly rejects heuristic attribution: "Detecting AI code is an anti-pattern."

6. **Playwright has become the evidence layer for UI agents.** Playwright v1.56 (October 2025) shipped native Planner/Generator/Healer agents. Trace Viewer captures "the action timeline, DOM snapshots, filmstrip, source locations, logs, console output, network traffic, metadata, and attachments" per step. Playwright MCP gives agents structured accessibility-tree views of running apps; GitHub Copilot Coding Agent has bundled Playwright MCP since July 2025.

7. **Semantic diff tooling is mature enough to replace text diffs.** diffsitter (tree-sitter AST diffs, 12+ languages), GumTree, SemanticDiff, and especially **RefactoringMiner-based AST diff** (Alikhanifard & Tsantalis, *A Novel Refactoring and Semantic Aware Abstract Syntax Tree Differencing Tool*, ACM TOSEM 2024 / arXiv 2403.05939) — which fixes GumTree's lack of multi-mapping, semantically incompatible AST node matches, language-clue ignorance, refactoring blindness, and lack of commit-level diff support — outperform line diffs significantly on an 800 bug-fix + 188 refactoring commit benchmark.

8. **Visual regression is solved at the platform level.** Percy (DOM snapshot + Visual Review Agent, real device cloud), Chromatic (Storybook-native, TurboSnap, git-aware baselines using merge-base), Applitools (Visual AI with Ultrafast Grid, floating/ignore regions, AI Match Levels) are the credible enterprise platforms.

9. **AI code review tooling is maturing with measured tradeoffs.** Per Macroscope's published Code Review Benchmark (118 self-contained runtime bugs across 45 open-source repositories): Macroscope 48%, CodeRabbit 46%, Cursor BugBot 42%, Greptile 24% (on Macroscope's set; Greptile reports 82% on its own 50-bug benchmark), Graphite Diamond 18%. CodeRabbit averages 10.84 comments per PR (only 4.69 runtime-relevant); Cursor BugBot averages 0.91 comments per PR, nearly all runtime-relevant. Independent reviewers behave as evidence-generators, not second AI authors.

10. **Risk and confidence must be communicated without false trust.** Veracode's Spring 2026 GenAI Code Security Update found the gap between "security pass rates…approximately 55%" and "near-perfect syntax correctness rates exceeding 95%" — the mismatch that produces over-confidence. The right pattern is to anchor confidence to *evidence vectors* (passing tests, accessibility, security scans, dependency audits) rather than to model self-reports.

## Details

### Section 1 — Core Problem
Multi-agent coding compounds three independently-hard problems: non-determinism, hidden state, and emergent multi-agent behaviors. A debuggable system must answer: *Which agent changed this line, in which session, under what prompt? What did the agent see (files, docs, errors)? Why this approach over alternatives? What tests were run, skipped, or written? What is confidence, and on what evidence? What is the blast radius if I revert? Is this duplicate of work another agent did? Did two agents make conflicting assumptions? Did this UI element change because of this commit, or did a side-effect ripple in?*

Concrete failure modes in 2024–2026 production deployments: overlapping edits in shared files when worktree isolation isn't enforced; silent regressions because tests pass but don't assert changed behavior; hallucinated fixes — invented APIs, stray imports (arXiv 2509.22040, "Your AI, My Shell," documented attack-success rates up to 84% for prompt-injection-driven malicious commands in Copilot/Cursor); context drift as summarization compresses prompt history; "approval churning" where agents resubmit unresolved changes (MSR 2026, arXiv 2601.00753, showed high-effort PRs are predictable at creation time from structural footprint metrics with AUC > 0.95, intercepting two-thirds of expensive PRs at a 20% review budget).

### Section 2 — Visual Explainability for Code and UI Changes
Effective interfaces are layered, not monolithic:
- **Agent color overlays on rendered UI**: each agent gets a hue; hovering a button reveals (agent, session, prompt, commit). A hidden inspection layer toggles like Chrome DevTools but with AI provenance instead of CSS rules.
- **"Why did this change?" panels** anchored to a file, route, or UI element; combine DOM provenance with AST diff and the agent transcript.
- **Timeline playback / scrubbable filmstrip** of UI evolution — modeled on Playwright Trace Viewer's Before/Action/After snapshots.
- **Agent-specific heatmaps**, **conflict overlays**, and a **"hidden consequences" mode** highlighting accessibility tree, ARIA, focus order, and side-effect API call changes invisible in the diff.

These belong in *all* of: IDE (Cursor/VS Code), browser preview, PR review (CodeRabbit/Macroscope post), CI report (Playwright HTML), and a dedicated AI debugging console (Devin Session Insights, Langfuse trace explorer) — driven by consistent provenance metadata flowing through all of them.

### Section 3 — State Representation and Inspectable Hidden Layers
A canonical "agent state JSON" composes three layers.

**Layer A — Identity & Task (OpenTelemetry GenAI standard)**: `gen_ai.agent.id` (e.g., `asst_5j66UpCpwteGg4YSxUnt7lPY`), `gen_ai.agent.name` (e.g., "Frontend Refactorer"), `gen_ai.agent.version`, `gen_ai.conversation.id` (e.g., `conv_5j66UpCpwteGg4YSxUnt7lPY`), `gen_ai.operation.name` (`invoke_agent` | `create_agent` | `invoke_workflow`), `gen_ai.tool.type` (Extension | Function | Datastore).

**Layer B — Execution State (event-sourced; OpenHands V1 / LangGraph)**: append-only stream of `Action` / `Observation` events emitted via `on_event(event)` callbacks. OpenHands V1 (arXiv 2511.03690): "agents emit structured events… through callbacks, separating event generation from execution control. This design enables: (1) security interleaving — actions can be reviewed or blocked before execution based on risk analysis; (2) incremental execution — the agent advances one step at a time, supporting pause/resume, recovery from context overflows, and condensation for long conversations; (3) event streaming — intermediate results are emitted in real time for UI updates and monitoring." Reasoning is persisted as `reasoning_content` / `thinking_blocks` on ActionEvent/MessageEvent so they're replayable. LangGraph `StateSnapshot`: `{v, ts, id, channel_values, channel_versions, versions_seen, parent_checkpoint_id}`.

**Layer C — Decision Context (the "why")**: files read/modified (with hashes), commands run, browser actions, screenshots; tests selected/skipped, errors observed and recovered from; hypotheses considered and rejected (summarized, not raw CoT); confidence (calibrated), risk class, blast-radius signals; assumptions made because of missing context; memory used (which microagent/skill triggered, which doc retrieved), external URLs read, prompt context, token budget; user instructions interpreted vs ignored; safety constraints active; rollback points (checkpoint IDs).

State diffs operate **beyond file diffs**: diffing the *event stream* between two runs of the same agent (or between agent and human) reveals divergence in approach, not just text. Representation: structured JSON in cold storage (Postgres/ClickHouse), graph DB (Neo4j or typed graph) for causal links, time-series store for metrics. AG-UI Protocol's event types (`RunStarted`, `STATE_SNAPSHOT`, `STATE_DELTA`, `TEXT_MESSAGE_*`, `TOOL_CALL_*`, `REASONING_*`) provide a standard wire format for streaming this to UIs.

### Section 4 — Multi-Agent Provenance and Ownership
The "change provenance ledger" records, for every change at every granularity, who/what/why/when:

| Granularity | Recorded by | Storage |
|---|---|---|
| Commit | git author + agent_id in commit trailer | git history |
| Line | git blame + git-notes overlay | `refs/notes/ai` (Git AI / Agent Blame schema) |
| AST node | AST diff with attribution per node | side DB joined by commit SHA |
| Function/Component | derived from AST + component map | search index |
| Test | test file metadata + author agent | CI + git notes |
| UI element | DOM stable ID → source map → file → commit | browser overlay |
| Dep / Migration / Config / Docs | file-pattern + agent_id | git notes |
| Decision | event log entry referencing prompt | event store |
| Tool call | OpenTelemetry span | trace backend |
| Runtime effect | distributed trace correlation_id | APM |

Git AI's open standard is the right architecture: provenance lives in git so it travels with the code, is offline-inspectable, and degrades gracefully ("When notes aren't available, the UI degrades to 'unknown' rather than guessing"). Example payload schema:

```json
{ "lines": { "1": "66392557c1f4b03f", "132": "66392557c1f4b03f",
             "253-259": "66392557c1f4b03f" },
  "prompts": { "66392557c1f4b03f": {
    "agent_id": { "tool": "cursor", "id": "a48660d5-...",
                  "model": "claude-4.5-opus-high-thinking" },
    "human_author": "...", "messages": [...] }}}
```

Attribution **should not be inferred** by heuristic; it should be recorded by the agent at write time, with "unknown" otherwise.

### Section 5 — Agent Communication and Collaboration Debugging
AGDebugger (CHI 2025) is the most rigorous published treatment. Its three primitives — (1) browsable agent conversation timeline, (2) interactive reset/edit of agent messages with checkpoint replay, (3) overview visualization for navigating message histories — map directly to UX patterns every multi-agent coding system needs. "Each reset forks the current conversation and creates a new conversation session, represented as a new column" — a column-per-fork pattern that scales to dozens of branches.

For coding specifically, layer on:
- **Role separation** (planner / builder / tester / reviewer / debugger / critic). OpenHands trained a critic model (Qwen 2.5 Coder Instruct 32B with TD-learning) to score candidate solutions — the structural pattern for disagreement tracking.
- **Conversation timeline** with collapsed intent+outcome badges by default, drill-down to full transcript.
- **Task handoff records** as explicit spans (OpenAI Agents SDK semantics).
- **Disagreement card** in `/best-of-n` with each side's evidence.
- **Duplicate-work detection** via intent hashing (file + AST subtree + diff size).
- **Context-drift detection** via embedding diff between turns.
- **Evidence-quality scoring**: each agent claim graded by whether tests verify it.
- **Escalation to human**: structured `OpenQuestion` event with mandatory fields, routed to Slack/IDE inbox.

### Section 6 — Visual and Semantic Testing for AI Coding Agents
Playwright is the spine. The trace.zip from a single test run contains a richer evidence package than most "agent debugging" tools produce natively.

- **Trace Viewer tabs** answer specific questions: Actions ("what did the test try?"), Source ("where in the test?"), Call ("locator, strict-mode, keys, timing"), Log ("what was Playwright waiting on?"), Console + Network, Metadata, Attachments.
- **Recording strategy**: `trace: 'retain-on-failure'` is the default best practice; in CI, also capture on first retry.
- **"Copy as Prompt" / "Fix with AI"** formalize the human → AI handoff for failing tests.
- **Playwright MCP** + Skills give agents structured accessibility-tree views, reducing token cost and improving locator quality.
- **Playwright v1.56 agents** (Planner / Generator / Healer) ship a complete inner loop.

Every PR should attach: before/after screenshots of touched routes; trace.zip for each test step that touched changed code; accessibility tree diff; visual diff (Percy/Chromatic/Applitools) keyed by component. Then map "changed UI element → which agent → which test asserts it." UI regions with no asserting test get a red "untested change" badge.

Detection patterns that matter: *tests that pass but don't assert changed behavior* (diff assertions pre/post); *overfitting to tests* (require an isolated tester/critic when one agent writes both impl and tests); *missing negative tests / edge cases* (coverage on changed branches + LLM-as-judge); *accessibility regressions* (Axe/Deque + AOM-tree diff).

### Section 7 — Semantic Diffs Beyond Normal Git Diffs
Substitutes for text diffs: AST diff (diffsitter, GumTree, astdiff, RefactoringMiner); type-aware / data-flow / control-flow diffs from compiler IR; API contract diffs (oasdiff, `buf breaking`); schema diffs (atlas / sqitch / dbmate); route diffs; component-tree diffs (react-docgen / Storybook); state-machine diffs (xstate); dependency-graph diffs with CVE delta; prompt/config/feature-flag diffs as first-class; performance / accessibility / security / a11y / i18n / telemetry diffs as their own panels.

**Diffs grouped by intent** is the killer UX. Use a critic LLM over the AST diff + test changes to label intent:

| Intent | Visual treatment |
|---|---|
| Bug fix | Green outline; link to failing test now passing |
| Refactor | Blue; suppress noise; "behavior-preserving" badge with evidence |
| Test addition | Light purple; coverage delta |
| UI copy change | Yellow; before/after string diff |
| Layout change | Orange; visual diff thumbnail |
| State-management change | Pink; state-machine diff |
| API contract change | Red; compatibility delta |
| Dependency update | Gray; vulnerability scan delta |
| Generated boilerplate | Dimmed; collapsed by default |
| Speculative experiment | Striped border; mandates human review |
| Rollback / Conflict resolution | Black / Diamond marker; cross-reference |
| Human vs Agent | 👤 vs 🤖 icon |

### Section 8 — Replay, Rollback, Branching, and Time-Travel
Three complementary backends:
- **Git** for code-level rollback (not sufficient for stateful work).
- **Event sourcing** for agent-level replay — append-only event log enables deterministic re-execution. LLM responses must be substituted verbatim at replay; model identifier, decoding params, and tool versions must be recorded.
- **Container/database snapshots** for environment-level rollback. Devin's blockdiff VM snapshot format is the production reference. Cognition (Sept '24 product update) describes the "restore checkpoint" affordance: "Scrub Devin's timeline and click the 'restore checkpoint' icon at the bottom right corner. This restores Devin to a previous point in time, rolling back Devin's files and memory."

Full pattern stack: step-by-step replay; checkpoints every N steps + on tool-call boundaries; branch- and sandbox-per-agent via worktrees + Docker; automatic savepoints before risky ops (migrations, dependency updates, deletions); rollback granularity from single action up to entire session; side-by-side alternative solutions (Cursor `/best-of-n`); merge-by-AST-subtree from competing branches; causal-chain preview before rollback; warning when rollback affects dependent changes. CRDTs / operational transforms are unnecessary for code (git suffices); they're relevant only for collaborative editing of *transcripts* and *plans*.

### Section 9 — Observability for AI Coding Agents
Data model:
- **Spans** for user request, agent task, agent step, sub-task, tool call, file read/write, test run, browser action, reasoning summary, human intervention.
- **Correlation IDs** flowing user_request_id → task_id → agent_id → session_id → commit_sha → test_run_id → trace_id → deployment_id.
- **Metrics**: productivity (PRs/week, accepted-LOC, durability over 30/60/90 days), reliability (test pass rate, regression rate per merged PR), cost (tokens, ACUs, $ per successful task), latency (TTFB per step), tool-call failure rate, prompt-injection trip rate.
- **Error budgets** per agent and per change-type.
- **Regression attribution**: walk correlation_id from production alert back to prompt.

OpenTelemetry GenAI provides the wire format. Content is opt-in (`OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT=true`). Coding-specific extensions: `gen_ai.code.files_modified`, `gen_ai.code.tests_run`, `gen_ai.code.commit_sha`, `gen_ai.code.pr_number`, `gen_ai.code.diff_intent`, `gen_ai.code.risk_class`.

Stack: **Langfuse** (MIT, ClickHouse-backed; ClickHouse acquired Langfuse on January 16, 2026, per ClickHouse's official blog: "We are thrilled to announce that ClickHouse has acquired Langfuse, the leading open-source platform for LLM observability, evaluations, and prompt management") or **Arize Phoenix** (OTel-native) for the observability spine; **LangSmith** if LangChain/LangGraph-native; **Helicone** as a drop-in proxy for cost only; **Braintrust** for evaluation/regression blocking. Pair with whole-stack APM (Datadog / Honeycomb / New Relic) for infrastructure correlation.

### Section 10 — Human Review and Code Review UX
The PR view for multi-agent work needs more, not less, surface area than human PR review:
- **Agent-authored summary** with a "show me the receipts" toggle — every claim links to evidence (passing test, file changed, doc consulted).
- **Intent-grouped diff** (Section 7) as default view.
- **Risk card** (low/med/high) with blast radius, test coverage, critical-path touch, security relevance, dependency changes, migration impact.
- **Confidence card** as an evidence *vector* across tests / accessibility / security / perf / reviewer — never a single number.
- **Trace links** to agent timeline; **test-evidence cards** with screenshots/trace; **visual-diff cards**; **provenance badges** per file.
- **Warnings**: unreviewed assumption, unsupported claim, security-sensitive, dependency risk, generated-code indicator, hot-spot file (frequently rewritten by AI).
- **Reviewer-question suggestions** by change type ("This touches auth — confirm session invalidation on logout").
- **Filters** by agent, task, risk, file type, UI region, test coverage, confidence, intent.
- **Anti-summary-hiding**: AI summaries are additive, never substitutive; expand-by-default for high-risk diffs; post-merge trust-calibration tracking.

Production tools: CodeRabbit (raised a $60M Series B led by Scale Venture Partners with NVentures/NVIDIA Venture Capital on September 16, 2025, at a $550M valuation, total funding $88M, per TechCrunch; broadest platform coverage, somewhat noisy); Greptile v3 (full-codebase code graph, multi-hop investigation, 82% catch rate on Greptile's own 50-bug benchmark); Cursor BugBot (low noise, Autofix with Cloud Agents launched February 2026 with ~35% of proposed changes merged directly); Graphite Diamond (low false-positive rate); Qodo Merge (multi-agent, open-source PR-Agent core); Macroscope (usage-based pricing, deep AST analysis for 8 languages).

### Section 11 — Debuggable Prompt, Memory, and Context Management
A "Prompt + Context Store" must be a first-class artifact, queryable per session:
- *What user instructions were used vs ignored*: an `interpreted_instructions[]` array per turn, tagged `applied` / `deferred` / `dropped` with reason.
- *Which system constraints were active*: policy IDs at each step.
- *Which memories/microagents were retrieved*: in OpenHands V1, "Skills can be defined programmatically or loaded from markdown files (`.openhands/skills/`, or compatible formats like `.cursorrules`, `agents.md`)" — each load is an event.
- *Files included/excluded*: with size and inclusion reason.
- *External docs read*: URLs + content hash + retrieval method.
- *Previous errors remembered*: linked event IDs from prior sessions.
- *Stale-memory detection*: cache freshness against file modtimes.
- *Hallucination markers*: facts asserted without retrieval citations get flagged.
- *Context drift across agents*: diff context-window contents at handoff.
- *Summarization audit*: store both original tokens and the summary with a similarity score.
- *Internal evidence citation*: agents output structured `<evidence src="…file:line"/>` markers binding every code suggestion to its source basis.

### Section 12 — Conflict Detection and Resolution
Conflict types and detection:

| Type | Detection | Visualization |
|---|---|---|
| File edit | git merge conflict | 3-way diff |
| Semantic (same intent, different impl) | AST similarity + intent classifier | Side-by-side AST diff |
| UI | DOM stable ID match + visual diff | Overlay both renders |
| Test | name / assertion target match | Test list with both versions |
| Architecture | dependency-graph divergence | Graph diff with red edges |
| Dependency | lockfile clash | Resolution table |
| Security | one agent removes a check, the other adds | Highlight in security card |
| Performance / Accessibility / Product req | bench / a11y tree / req matrix | Side-by-side / tree diff / traceability |
| Naming / State-mgmt / Routing / Schema / Prompt / Memory | symbol resolution / FSM / route / migration replay / prompt diff / memory log | Specialized diff |

The visual conflict view should always show *evidence supporting each side*: passing tests, the prompts that drove the decision, time of decision, agent identity. A "diff of diffs" view comparing two agents' attempts against the base is the right primitive.

### Section 13 — Risk, Confidence, Uncertainty, Verification
Communicate uncertainty as a *vector of evidence*, not a scalar. Dimensions:
- **Test verification**: were changed lines exercised by a passing test? Was a new assertion added that would have failed before?
- **Accessibility**: Axe / aria-snapshot pass?
- **Security**: SAST / Semgrep / dependency scan clean?
- **Performance**: bench / Lighthouse pass thresholds?
- **Reviewer**: did a critic agent (Greptile-class) approve?
- **Human**: did a senior engineer click "looks good"?

Each dimension gets ✅ verified, ⏸ N/A, ⚠️ inferred, ❌ unverified. Aggregate confidence surfaces only when a minimum bar is met. The Veracode Spring 2026 data is the moral argument against single-number scores.

Risk classes:
- **Low**: behavior-preserving refactor, isolated component, full test coverage on touched lines, no dep change.
- **Medium**: new behavior on non-critical path, partial test coverage, doc/config change.
- **High**: critical-path touch, auth/security relevance, migration/dep change, low test coverage, large blast radius. High-risk gates merge: explicit human approval, no autoreviewer-only merges.

### Section 14 — Accessibility, Security, Privacy, Compliance Debugging
Diff types every PR needs:
- **Accessibility tree diff**: serialize the AOM before/after; ARIA / role / name diffs.
- **Keyboard navigation trace**: replay Tab + Enter; show focus order pre/post.
- **Color contrast diff**: WCAG checks per text element.
- **Screen-reader impact summary**: human-readable diff of what changed for AT users.
- **Security taint tracking**: dataflow from user inputs to sinks; flag new untrusted paths.
- **Permission boundary diffs**: new public endpoints flagged.
- **AuthN/AuthZ path diffs**: middleware order changes, new bypass routes.
- **Secret leakage detection**: trufflehog / gitleaks on PR.
- **Prompt-injection exposure**: LlamaFirewall (Meta, arXiv 2505.03574) with PromptGuard 2, Agent Alignment Checks, CodeShield is the open reference.
- **Dependency CVE delta**, **PII handling diff**, **audit trail export to SIEM**, **data-retention TTL diffs**, **cookie/telemetry/analytics diffs**.

### Section 15 — Product Design: Ideal Debuggable Multi-Agent Coding Environment
Information architecture: a single workspace that orbits the **PR-as-investigation-object**.

Primary panels (always visible): code editor with inline agent attribution gutter + change-intent heatmap; browser preview with a hidden inspection layer toggleable for agent provenance per element; visual change overlay colour-coded by agent.

Inspectors (collapsible): hidden AI provenance layer; structured state JSON inspector; agent timeline column-per-agent AGDebugger-style; task DAG of subtasks with dependencies and parallelism; change provenance ledger (sortable table); semantic diff viewer (intent-grouped); embedded Playwright trace viewer; test-coverage overlay coloring UI regions by coverage status.

Operational consoles: rollback & replay console (event-stream slider, restore-checkpoint); risk dashboard (risk-class grid + evidence vectors); human review workspace; conflict-resolution view for /best-of-n; observability trace explorer; memory & context inspector; deployment-readiness report composite (merge/block recommendation).

Key user flows:
- *Investigate a regression*: production alert → trace → commit → PR → agent session → prompt → re-run replay with edited prompt.
- *Compare two agent solutions*: /best-of-n. Diff-of-diffs view; pick winner; cherry-pick from loser.
- *Approve high-risk PR*: open risk dashboard; for each ⚠️ dimension, drill to evidence; mark assumptions reviewed; merge.
- *Roll back an agent's contribution*: select agent in provenance ledger → preview blast radius → confirm.

### Section 17 — Evaluation Framework
Metrics: time-to-understand-a-change (instrumented review UI); time-to-identify-regression-cause; time-to-safely-approve-a-PR per risk class; hidden regressions caught pre-merge by AI reviewer; unsupported agent claims caught; trust calibration via post-merge audit; reduction in unnecessary review time; reduction in duplicated agent work; reduction in failed merges, flaky tests, rollback rate; rollback speed; test coverage of changed behaviors (not just changed lines); attribution accuracy via sample audits; explanation accuracy via LLM-judge + spot human grading; cognitive load (NASA-TLX or equivalent); false-confidence rate (high-confidence PRs that broke production).

Benchmark experiments:
- *Task set A*: SWE-Bench Verified subset. Compare default Cursor/Claude Code vs same + debuggability suite. Primary metric: time-to-approve-PR + regression-rate one month after merge.
- *Task set B*: deliberate injection of bugs into AI PRs; measure pre-merge catch rate by tool.
- *Task set C*: long-horizon SWE-EVO tasks; measure cause-of-failure attribution accuracy.
- *Within-subjects user study*: 14+ engineers (à la AGDebugger), counterbalanced ordering, debug a real multi-agent run with vs without the inspection layer.

### Section 18 — Implementation Architecture

**MVP (4–6 weeks for a small team)**:

```
Agent runtime (OpenHands / Claude SDK / LangGraph)
   ↓ emits OpenTelemetry GenAI spans + custom gen_ai.code.* attrs
   ↓                                      ↓
Langfuse / Phoenix (OTel ingest)    Agent commits to git
                                    + writes refs/notes/ai (Git AI schema)
                                          ↓
                       GitHub Action augments PR:
                         • pulls trace from Langfuse
                         • reads git notes for provenance
                         • runs Playwright + uploads trace.zip + screenshots
                         • runs AST diff (diffsitter / RefactoringMiner)
                         • posts risk + evidence comment
                         • blocks merge unless required checks pass
```

Stack: Postgres / ClickHouse (events), MinIO / S3 (traces, screenshots), Playwright in CI (evidence), GitHub Actions (PR augmentation), Git AI / Agent Blame open standard (notes schema). Agents run in `--worktree` mode. Branch protection requires the PR augmentation check to pass.

**Advanced architecture (6–12 months)**: add event-store backend (EventStoreDB / Kurrent or Postgres + outbox) keyed on session_id, supporting deterministic replay; graph DB (Neo4j) for causal links; source-map service mapping component → (source_file, line_range, commit, agent); DOM stable-ID injection at build time; browser overlay extension reading from source-map service; visual diff service with Percy / Chromatic / Applitools webhooks; semantic diff service (RefactoringMiner / GumTree); critic-model gate (OpenHands TD-trained); conflict-resolution UI for /best-of-n; replay service re-executing edited sessions; containerised sandboxing with VM snapshots (Devin blockdiff-style or LXD/Firecracker); LlamaFirewall as PR gate; per-tenant key separation; per-artifact TTL; right-to-be-forgotten flow.

Security/privacy: prompt content opt-in only; encrypted at rest; per-tenant key separation; minimum-scope tool tokens per agent; agents identified with distinct service accounts (no shared creds).

Performance trade-offs: event-stream writes add milliseconds per step (negligible vs LLM latency); full trace.zip per test adds MBs of CI artifacts but is the highest-ROI debugging artifact.

### Section 20 — Final Recommendations

**Must-have** (do not ship multi-agent coding without these):
- Git worktrees per agent for filesystem isolation.
- Line-level AI provenance via git notes (Git AI open standard).
- OpenTelemetry GenAI tracing of every agent invocation and tool call.
- Playwright trace.zip + before/after screenshots attached to every PR touching UI.
- Risk-class banner on every PR; high-risk requires named human approver.
- Critic / independent AI reviewer (Greptile or equivalent) as a separate check, not the same agent that wrote the code.
- One-click rollback per agent.

**Should-have (within 6 months of shipping)**:
- Event-sourced agent state with deterministic replay (OpenHands V1 pattern).
- Semantic diff grouped by intent, replacing text diff as the default view.
- Evidence-vector confidence cards (no single confidence numbers).
- AGDebugger-style agent timeline with checkpoint reset.
- Visual regression integration (Percy / Chromatic / Applitools).
- Accessibility tree diff.
- Prompt + Context store with memory audit.
- LlamaFirewall-class prompt-injection / unsafe-output detection.

**Advanced (within 12 months)**:
- Source-map + DOM stable-ID + browser overlay for live UI provenance.
- Conflict-resolution view for /best-of-n outcomes.
- Causal dependency graph for blast-radius analysis.
- Critic-model gate (TD-trained).
- Deployment-readiness composite dashboard.
- VM snapshot / blockdiff per session.

**Experimental (research bets)**:
- Multi-agent "agent responsibility matrix" learned over time.
- Trust-calibration tracking and adaptive review depth.
- Cross-codebase provenance (an agent's track record carries between repos).
- AI-judged "explanation quality" scoring of agent summaries.

**Roadmap**:
- *First 30 days*: adopt git worktrees per agent. Install Git AI (or equivalent) for git-notes-based provenance. Stand up Langfuse or Phoenix and wire OpenTelemetry GenAI spans from your agent runtime. Require Playwright traces in CI on PRs touching UI code. Add a manual risk-class field to every PR template.
- *First 90 days*: ship the PR-augmentation bot (intent-grouped diff, evidence cards, risk class). Integrate an independent AI reviewer (Greptile / Macroscope / Cursor BugBot). Add visual regression (Percy or Chromatic). Add accessibility tree diff. Build the per-agent timeline view (start read-only).
- *First 6 months*: build the event-sourced replay backend (OpenHands V1 reference). Add reset/edit on agent messages (AGDebugger). Ship semantic diff with intent grouping. Integrate prompt-injection scanning (LlamaFirewall). Build the conflict-resolution view for /best-of-n.
- *First 12 months*: ship source-map + DOM stable-ID + browser overlay. Build the composite deployment-readiness dashboard. Add VM snapshots / blockdiff for full session rollback. Train a critic model on your codebase. Begin trust-calibration experiments and tune review depth adaptively.

**Decision thresholds** (when to escalate to the next tier):
- If PR review time has not dropped 30% within 90 days, the augmentation bot's signal-to-noise is too low — invest in tuning before adding features.
- If rollback rate after merge exceeds 5% on AI PRs, gate high-risk PRs harder and improve test-evidence requirements.
- If two-agent conflicts occur in >10% of /best-of-n runs, the conflict-resolution view jumps from advanced to must-have.
- If prompt-injection trip rate >0 in 90 days, LlamaFirewall-class scanning jumps to must-have immediately.

## Recommendations

1. **Start with provenance and worktrees, not a new agent framework.** The ROI of git-notes-based AI attribution + worktree isolation in week 1 dwarfs any framework migration. Use the agents you already have; instrument them.
2. **Adopt OpenTelemetry GenAI conventions early.** Even if you self-host on Langfuse or Phoenix today, OTel is the durable spine. Every vendor lock-in escape hatch flows through it.
3. **Treat the PR as the inspection object.** All evidence (traces, diffs, screenshots, accessibility, risk) collapses into the PR. Resist building separate "agent debuggers" — they fragment attention.
4. **Make the independent reviewer agent independent.** Greptile's principle: "Creation and evaluation are different jobs, and the reviewer works best when it is independent." Do not let the same model write and approve.
5. **Anchor confidence to evidence, not model self-reports.** A single confidence number is worse than no number. Use vectors (tests / a11y / security / perf / human) and require explicit ❌ on missing dimensions.
6. **Invest in event sourcing once, benefit forever.** Replay, time-travel debugging, conflict-resolution, training data for critic models, and audit trails all share one substrate.
7. **Make rollback trivial and visible.** 1-click per-agent rollback + blast-radius preview is more reassuring to humans than any safeguard short of formal verification.
8. **Default to /best-of-n + critic gate on high-risk paths.** OpenHands' SOTA on SWE-Bench was achieved this way; the same pattern lifts production reliability.
9. **Measure cognitive load, not just throughput.** Productivity gains that double review burden are a net loss. Track time-to-understand-a-change as the primary north-star metric.

## Caveats

- **Pace of change**: Cursor 3 shipped major features in April 2026; OpenHands V1 in November 2025; Playwright agents in October 2025; ClickHouse acquired Langfuse in January 2026; CodeRabbit closed its $60M Series B in September 2025. Any specific tool recommendation may be displaced within 6 months. Architecture choices that depend on open standards (OpenTelemetry GenAI, git notes, Playwright traces, OpenAPI) age better than tool choices.
- **Benchmark numbers are mostly self-reported.** Greptile's "82% catch rate" is on its own 50-bug benchmark; Macroscope's 118-bug benchmark gives Greptile 24% under their methodology, with Macroscope 48%, CodeRabbit 46%, Cursor BugBot 42%, Graphite Diamond 18%. Treat detection percentages as directional, not absolute, and trust independently reproduced benchmarks more than vendor-reported ones.
- **OpenTelemetry GenAI agentic conventions (tasks / teams / artifacts) are still in proposal stage** (issue #2664); the stable conventions cover agent / tool / conversation but not yet multi-agent collaboration semantics. Expect breaking changes.
- **Devin internal details** (blockdiff snapshot timing, MultiDevin specifics) come partly from secondary sources; primary Cognition posts confirm the user-facing capability, but specific implementation timings should be verified before external citation.
- **Veracode's Spring 2026 GenAI Code Security Update finding** ("security pass rates have remained stubbornly flat at approximately 55%…near-perfect syntax correctness rates exceeding 95%") is on a benchmark of generation tasks, not a measurement of code that survived review; real-world post-merge incident rates from AI code are not yet well-published.
- **Speculative claims to watch**: some vendor pages assert "review-time dropped 80%" or "merges in under 3 minutes." Several reviewed sources are clearly future-tense or marketing-toned and should be treated as marketing until independently reproduced.
- **Prompt-injection risk in coding agents is real and rising.** arXiv 2509.22040 and 2601.17548 document attack-success rates in the 60–84% range on Copilot and Cursor when external resources are poisoned. Section 14's recommendations should be treated as urgent for any internet-connected coding agent.
