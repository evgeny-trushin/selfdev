# Debuggable Multi-Agent AI Coding

The central product problem in multi-agent AI coding is no longer “can the agents write code,” but “can a human reconstruct causality across code, UI, tests, browser state, prompts, and runtime effects quickly enough to trust, debug, govern, and, when needed, reverse what the agents did.” Current systems already prove parts of this stack are feasible: multi-agent frameworks coordinate specialized roles, coding agents can work in isolated branches or sandboxes, browser tools can capture step-by-step traces, modern devtools can map runtime activity to components, visual testing systems can cluster UI diffs, and observability systems can correlate traces, logs, and metrics. What is still missing is a unified provenance and debugging model that ties all of those signals together around one question: **why did this visible change happen, who caused it, and what evidence supports it?** citeturn10search0turn10search1turn10search2turn10search3turn26view0turn12search1turn27view0turn30view3turn4search13turn31view0

## The debugging problem and the questions humans need answered

Multi-agent coding is hard to debug because the system’s state is split across many layers that evolve at different speeds: repository state, browser state, test state, build state, prompt state, retrieved memory, and agent-to-agent coordination state. Multi-agent research systems such as AutoGen, MetaGPT, and ChatDev explicitly rely on multiple interacting roles and message exchanges; SWE-agent shows that performance changes materially when the agent-computer interface changes; and current production tools such as GitHub Copilot cloud agent, Cline, Claude Code, and OpenHands all expose different slices of the execution path rather than a single causal model. The result is a larger search space for blame and a greater chance that the explanation available to a reviewer is only a lossy summary of what actually happened. citeturn10search0turn10search1turn10search2turn10search3turn26view0turn26view9turn26view7turn13search8

Several failure modes recur for first-principles reasons. Overlapping edits happen because parallel agents work on related files, branches, or worktrees at once; duplicated work happens because planners and implementers often have only partial awareness of each other’s context; stale context happens because memory systems, summaries, and selective retrieval can omit or distort facts; silent regressions happen because a small code diff can trigger a larger UI or runtime effect; and non-deterministic behavior happens because external calls, browser interactions, clocks, caches, and model outputs are not replay-stable by default. Temporal’s workflow model is a useful contrast: it requires deterministic orchestration and records an event history precisely because replay breaks when decisions are not recoverable from history; by design, non-deterministic work such as API calls belongs outside the replay path. AI coding systems need the same separation, but most current tools do not impose it yet. citeturn26view7turn22search23turn28view3turn32view2turn32view3turn32view4turn26view9

The specific human questions a debuggable multi-agent coding system must answer are not mysterious; they are traceability questions. Git already answers a narrow version of them at commit and line level with `git blame`, `git bisect`, `git notes`, and `git range-diff`; GitHub and CODEOWNERS answer a narrow version at repository and review ownership level; Playwright answers a narrow version for browser/test steps; and framework devtools answer a narrow version for rendered components. The gap is that these answers are fragmented and usually stop at one layer. citeturn34search4turn34search0turn34search19turn34search3turn34search1turn34search2turn27view0turn29view8turn29view7turn29view5

A good multi-agent debugging system should therefore treat the following as first-class queries rather than ad hoc forensics:

| Human question | Minimum evidence needed | Best answer form |
|---|---|---|
| Which agent changed this file, component, UI element, test, route, dependency, prompt, or config? | Agent ID, step ID, patch hash, file/component mapping, timestamp | Provenance card with drill-down |
| Why was the change made? | Task objective, cited evidence, failing test or issue link, reasoning summary | “Why this changed” panel |
| What intermediate state did the agent see? | File snapshot IDs, DOM snapshot, logs, tool outputs, prompt context refs | Replayable state snapshot |
| What influenced the change? | Browser actions, API payloads, terminal output, screenshots, docs consulted | Evidence bundle |
| Which changes are related? | Shared task ID, causal edges, same failing test, same UI region, same dependency | Change cluster / DAG |
| Which changes are speculative, verified, reverted, or untested? | Status state machine plus evidence links | Status badges with verification record |
| Which change introduced a regression? | Correlated span/test/deploy IDs, before/after traces, bisect or blame result | Regression attribution report |
| Which agents disagreed, and how was it resolved? | Message timeline, conflict object, final decision record | Collaboration timeline |
| Which user-facing behaviors changed, even if the diff is small? | Visual diffs, ARIA diff, network diff, analytics diff, feature-flag/runtime diff | Behavior change summary |
| Which changed surfaces lack adequate testing? | Changed-file/component graph plus test coverage and trace coverage | Untested-change warning |

The most important implication is architectural: **every meaningful agent action must become a queryable event with attached artefacts**. If the action is not queryable later, it is not really debuggable.

## What the current landscape already proves

The current tool landscape is surprisingly rich, but the capability is scattered. The strongest existing products solve one or two layers well and leave the cross-layer provenance problem unsolved. GitHub Copilot cloud agent can research a repository, create a plan, work in an ephemeral GitHub Actions environment, make code changes on a branch, run automated tests, and expose work through commits and logs; Copilot code review can leave PR comments and suggested changes, but its comments are explicitly non-blocking and can repeat on re-review. Cline already demonstrates two especially relevant ideas: checkpoints captured after each tool use in a shadow Git repository, and multi-agent Kanban orchestration with isolated git worktrees and dependency chains. Claude Code demonstrates inspectable memory loading, hooks, and sandbox/permission controls. OpenHands demonstrates sandboxed runtimes, real-time event streams, and machine-readable JSON output. Cursor exposes semantic and agentic code search, codebase indexing, and an Explore subagent. Together, these tools prove that auditability primitives can be made product-grade; they do not yet unify them into a coherent human debugging model. citeturn26view0turn26view1turn26view9turn12search1turn26view8turn26view7turn23search18turn13search6turn13search8turn22search1turn23search11

Frontend and browser tooling already prove the viability of a visual provenance layer. Playwright Trace Viewer and UI Mode provide step timelines, DOM snapshots, before/after views, network filtering, console filtering, action details, affected-test filtering, retries, videos, custom reporters, and accessible-tree snapshots. Chrome DevTools, React DevTools, Angular DevTools, and Vue DevTools already map runtime behavior to framework-level concepts, components, and source files; Angular’s new Chrome Performance panel integration is especially important because it shows that framework-specific semantic traces can live inside the browser’s native performance timeline and use color-coded entries to distinguish authored code from framework-generated work. Chrome DevTools MCP adds the idea that agents can use live browser inspection tools rather than static HTML scraping, which is exactly the right direction for debugging visibility. citeturn27view0turn27view1turn27view2turn27view3turn27view4turn27view5turn27view8turn28view0turn29view8turn29view7turn29view5turn29view6turn30view2turn30view3turn29view3turn29view4

Visual review platforms prove related ideas for UI debugging. Chromatic captures snapshots plus metadata and positions review around intentional-versus-unintentional UI change; Percy groups matching diffs automatically to reduce review overload; Applitools surfaces exact broken DOM elements through root-cause analysis; and BrowserStack Percy’s newer visual review agent points toward AI-assisted clustering and triage of snapshot changes. These are strong foundations for multi-agent UI explainability, especially for before/after screenshots, semantic grouping of visual changes, and “review only changed regions” workflows. citeturn4search4turn4search8turn4search13turn4search10turn4search14turn4search21

Observability and provenance systems solve a different but highly relevant part of the problem. OpenTelemetry provides a vendor-neutral model for traces, metrics, logs, context propagation, baggage, processors, and semantic conventions, including conventions for CI/CD, feature flags, GraphQL, and generative AI. OpenAI’s Agents SDK tracing already records generations, tool calls, handoffs, guardrails, and custom events. LangSmith shows that agent trajectories, online/offline evals, nested trace payloads, and trajectory queries are operationally valuable. OpenLineage shows how run/job/dataset lineage can be captured as standardized events. SLSA, in-toto, and Sigstore show how attestations, ordered steps, “who/what/when,” and transparency logs can be made tamper-evident. These systems are not coding-agent debuggers, but together they provide the conceptual vocabulary for one. citeturn31view0turn31view1turn31view2turn31view3turn31view4turn31view5turn31view6turn31view7turn31view8turn32view0turn32view1turn5search8turn5search1turn5search19

## A reference model for state, provenance, ownership, and collaboration

The most robust design pattern is to combine four established ideas into one software-agent-specific model: OpenTelemetry-style traces for causality, OpenLineage-style run/job objects for lineage, supply-chain-style attestations for tamper-evident provenance, and Git-style patches/blame/range-diff for source control history. In practice, that means a multi-agent coding platform should emit an append-only event stream; store immutable artefacts such as screenshots, traces, DOM snapshots, and test outputs; maintain versioned patch objects; and generate a graph index that links requests, tasks, agents, files, components, tests, UI regions, deployments, and rollbacks. This is the closest fit to the problem because event sourcing preserves intent and ordered state changes, while graph queries support causal questions such as “show all UI elements affected by the authentication refactor” or “show all changes made after the checkout failure appeared.” citeturn31view0turn31view2turn31view4turn32view0turn32view1turn21search2turn21search17turn5search8turn5search1

A canonical **agent state JSON** should be captured at every meaningful step, not just once per task. The schema should include the fields the user asked for because the existing tooling already shows they map cleanly onto real product surfaces: agent identity and role; task and subgoal; files read and modified; commands run; browser actions; screenshots; errors; hypotheses; tests selected or skipped; confidence and risk; assumptions; external documentation; prompt context and memory used; safety constraints or guardrails triggered; rollback points; and token/time/cost data. OpenAI Agents tracing, Cline’s event namespaces, OpenHands’ event model, Playwright `TestInfo`, Claude Code memory, and OTel baggage/semantic conventions collectively show that this data is both collectible and queryable. citeturn31view8turn26view8turn13search8turn27view7turn26view7turn31view6turn31view7

Ownership should also be represented at multiple levels because no single mechanism is enough. **Line-level attribution** is still best answered through `git blame`; **patch-series attribution** through branch/commit/range-diff; **human stewardship** through CODEOWNERS; **non-invasive metadata** through `git notes`; **component-level ownership** through framework runtime mappings and source maps; **tool-call attribution** through trace events; and **runtime-effect attribution** through correlated spans plus browser/test artefacts. State diffing must likewise go beyond text diffs. Established building blocks exist: Tree-sitter enables incremental syntax-tree updates; GumTree demonstrates fine-grained AST differencing; Difftastic and SemanticDiff show language-aware review; CodeQL provides data-flow and taint analysis; `openapi-diff`/`oasdiff` detect API contract differences; `prisma migrate diff` compares schema sources and can help resolve drift; and OpenFeature and LaunchDarkly provide standardized or auditable representations for feature-flag changes. A practical semantic diff system should consume these specialized analyzers and then **group changes by intent**: bug fix, refactor, generated boilerplate, state-management change, accessibility change, dependency update, migration, rollback, or speculative experiment. That is far more legible than a flat file list. citeturn14search4turn14search7turn14search5turn14search2turn6search0turn6search12turn25search0turn25search8turn25search1turn25search9turn25search7turn17search3

Inter-agent collaboration should be visible, but compressed. The right model is not “show me every token”; it is “show me the conversation timeline, the handoffs, the disagreements, the evidence quality, and the final decision, with drill-down when I ask.” Multi-agent frameworks already separate roles such as planner, builder, tester, and reviewer; OpenAI Agents tracing includes handoffs and guardrails; OpenHands exposes real-time events; and LangSmith shows that full trajectories can be evaluated at intermediate-step level. The missing layer is a **collaboration debugger** that summarizes each handoff into task intent, evidence references, open questions, and explicit confidence. That would make duplicate work, deadlocks, and contradictory assumptions detectible before they become merge conflicts or regressions. citeturn10search0turn10search1turn10search2turn31view8turn13search8turn33view1

## Visual explainability, semantic diffs, and test evidence

The best place to make AI-driven changes understandable is not one place but a layered surface model. The **browser preview** should be the primary surface for user-facing change inspection, because that is where a reviewer can see what actually changed. The **IDE** should be the authoring and drill-down surface for source, state, and semantic diffs. The **CI report** should be the evidence archive for traces, screenshots, and test artefacts. The **pull request interface** should be the governance surface where summaries, risks, reviewers, and approvals are managed. Existing tools already divide labor this way: Playwright owns rich execution traces, framework devtools own runtime/component inspection, visual testing platforms own image comparison workflows, and PR tools own review and sign-off. A debuggable multi-agent product should unify them rather than pick one. citeturn27view0turn27view1turn29view3turn29view8turn29view7turn4search4turn4search13turn26view1turn34search2

For **visual explainability**, the highest-value pattern is a toggleable hidden inspection layer over the running app. Framework tools already prove the feasibility of component highlighting, component-to-source mapping, and runtime tracks: React DevTools exposes Components and Profiler panels; Vue DevTools can inspect the DOM tree and show which component renders it, then open the corresponding source in the editor; Angular DevTools debugs and profiles Angular apps; and Angular’s Chrome Performance integration shows that custom colored framework traces can live in the browser’s native timeline. The next step is to augment each rendered element or component boundary with AI provenance metadata: agent ID, timestamp, related files, related tests, related issue, confidence, and reasoning summary. citeturn29view8turn29view5turn29view6turn29view7turn30view0turn30view2turn30view3

A practical visual explainability stack should include the following original recommendations, all built on existing primitives. **Agent color overlays** should identify who touched which UI region. **Effect-type colors** should distinguish direct source edits from indirect effects, generated code, refactors, config shifts, dependency changes, and test-only changes. **Hover annotations** should expose provenance cards. **Greyscale unchanged mode** should suppress noise during review. **Conflict overlays** should highlight UI regions touched by multiple agents or multiple tasks. **Consequence mode** should reveal hidden behavior changes such as ARIA labels, event handlers, analytics events, permission checks, validation rules, and network calls, even when the pixels barely move. This last capability is especially important because Chrome’s accessibility tree, Playwright ARIA snapshots, source maps, and network views already show that major user impact often hides behind tiny visual diffs. citeturn29view0turn29view1turn27view3turn29view2turn27view1turn27view0

For **UI and visual testing**, Playwright should be treated as a first-class provenance source, not just a pass/fail gate. Trace Viewer and UI Mode already provide a time-travelable record of each action, before/after DOM snapshots, filtered network and console logs, step timing, videos, and per-run trace retention that is explicitly useful for comparing a failing flaky run to a passing run. Playwright also provides tags and annotations, affected-test filtering in UI Mode, `TestInfo` attachments, built-in and custom reporters, screenshot comparisons, and ARIA snapshots represented as YAML accessibility trees. The best multi-agent systems should attach agent IDs, task IDs, and changed-component IDs directly into test metadata and artefacts so a reviewer can ask “which agent wrote or selected this test?” or “what exactly does this passing test prove about the changed UI?” and get a direct answer. citeturn27view0turn27view1turn27view2turn27view3turn27view4turn27view5turn27view7turn27view8turn28view1

The most valuable testing UX addition is a **coverage overlay on the actual UI**, not just in source gutters. VS Code already shows inline test coverage in the editor, and Playwright already knows which locators and steps executed during a test; combined with source maps, component mappings, and DOM snapshots, that data can be projected back onto the rendered app. The system should therefore color UI regions as **tested**, **partially tested**, or **untested** after each run; highlight changed regions with no corresponding trace or assertion; and warn when a test only exercises unchanged areas. This is the fastest way to expose two common AI-agent failures: “tests pass but do not assert the changed behavior,” and “the agent overfit the implementation to the test.” citeturn8search2turn8search22turn27view1turn18search17turn18search5

Accessibility, privacy, and security debugging need their own panels because these regressions are often invisible in screenshot diffs. Chrome DevTools already exposes the accessibility tree and ARIA properties of DOM nodes; Playwright documents page-wide accessibility testing through `@axe-core/playwright` and supports ARIA snapshots of the accessibility tree; axe-core returns machine-readable JSON violations; CodeQL documents both data-flow analysis and taint tracking; GitHub documents risks and mitigations for prompt injection in cloud agent workflows; and OWASP now treats prompt injection as an explicit top LLM risk. In a multi-agent product, those signals should become **diffable evidence cards**: accessibility-tree diff, focus-order diff, permission-boundary diff, taint-path diff, telemetry/cookie diff, secret-leakage warning, and prompt-injection exposure warning. citeturn29view0turn29view1turn28view0turn27view3turn6search2turn6search0turn24search13turn24search10turn24search2

## Collaboration debugging, replay, rollback, observability, and governance

The collaboration process itself has to be understandable, because a large share of multi-agent defects are really coordination defects. ChatDev, MetaGPT, and AutoGen all operationalize role separation and message passing; GitHub Copilot cloud agent explicitly supports custom agents; Cline Kanban supports dependency chains between tasks; and OpenAI Agents tracing records handoffs and guardrails. The right product abstraction is a **visible agent timeline** with summarized messages, explicit handoff records, disagreement objects, and responsibility matrices. Each handoff should answer four questions: what was requested, what evidence was supplied, what assumptions were left open, and what verification is now required. That is the minimum compression needed to make collaboration legible without drowning the reviewer in raw chat logs. citeturn10search0turn10search1turn10search2turn26view0turn12search1turn31view8

Conflict detection should be layered because not all conflicts are text conflicts. Git merge conflicts are the shallowest case. More consequential are **semantic conflicts** where two agents make individually plausible edits that jointly violate a behavior or invariant; **UI conflicts** where two changes affect the same visible region differently; **architecture conflicts** where one agent moves responsibilities contrary to another’s assumption; **test conflicts** where a newly added test codifies the wrong behavior; and **context conflicts** where memories, rules, or instructions disagree. Current tools already provide building blocks: worktree isolation lowers edit collisions; Graphite’s stacked diffs and Gerrit’s patch sets make alternative revisions and dependencies reviewable; Claude Code exposes instruction loading and conflict debugging through `/memory`; and GitHub’s CODEOWNERS makes stewardship explicit. A debuggable agent platform should compute conflict maps over files, AST nodes, components, routes, tests, and UI regions, then present each conflict with evidence on both sides and a human-readable decision log for the resolution. citeturn12search1turn20search0turn20search5turn26view7turn34search1

Replay and rollback are where many current products are closest to a breakthrough. Cline checkpoints show the value of cheap savepoints after each tool use. Git offers the mature primitives for rollback, compare, and blame through `diff`, `show`, `restore`, `bisect`, `range-diff`, and notes. LaunchDarkly change history and audit log show how runtime config changes can be inspected and rolled back with preserved workflows and approvals. Prisma `migrate diff` shows how schema history can be compared to actual database shape and used to resolve drift. Temporal shows the deeper orchestration lesson: durable replay works when the system retains an ordered event history and treats that history as the source of truth, while non-deterministic operations are isolated and not recomputed during replay. For AI coding, the right design is therefore: deterministic replay for orchestration and local transforms where possible, cached event-history replay for previous LLM/tool observations, and explicit warnings when a rollback may invalidate dependent later changes. citeturn26view9turn34search0turn34search3turn34search7turn34search8turn20search17turn17search3turn16search19turn17search19turn25search1turn25search9turn32view2turn32view4

Observability should be imported almost wholesale from distributed systems, with software-agent-specific semantics added on top. OpenTelemetry already provides the right envelope: traces for hierarchical task execution, logs for searchable event records, metrics for reliability/cost/productivity, context propagation for correlation, baggage for cross-cutting attributes, processors for filtering and enrichment, and semantic conventions for standard attribute names. OpenAI Agents tracing shows that generations, tool calls, handoffs, and guardrails are traceable. Cline proves that coding-agent-specific event namespaces such as `task.*`, `workspace.*`, `hooks.*`, and `worktree.*` are practical. LangSmith proves the operational value of trajectory queries, online evals, and failure clustering. A multi-agent coding platform should therefore emit spans for request, task, subtask, file read, file write, patch apply, command run, browser action, test execution, review comment, human intervention, deployment, and rollback, all tied together by a single correlation chain. citeturn31view0turn31view1turn31view2turn31view3turn31view4turn31view6turn31view7turn31view8turn26view8turn33view2

Risk and confidence should be expressed as **evidence states**, not as vibes. The worst UX pattern here is a single global confidence score that invites false trust. A better model is per-change evidence classes such as: _verified by deterministic test_, _verified by browser trace_, _verified by static analysis_, _observed only in runtime_, _inferred but unverified_, and _unsupported explanation_. Risk should be computed from blast radius, critical-path involvement, absence of tests, security sensitivity, dependency or migration impact, and whether the change affects auth, payments, or data retention. Existing systems already expose parts of this: LaunchDarkly keeps auditable change history for feature flags; OTel has feature-flag semantic conventions; GitHub Copilot cloud agent documents prompt-injection mitigations and automated security analysis of generated code; and OpenSSF guidance explicitly warns that AI code assistants can introduce security issues and unsafe dependencies. citeturn17search3turn17search19turn31view7turn24search13turn24search5turn5search23

Human review UX should therefore show: a concise agent-authored summary; an intent-grouped semantic diff; provenance badges; evidence links; test evidence cards; visual diff cards; unsupported-claim warnings; security-sensitive-path warnings; dependency-risk warnings; and tailored reviewer checklists. The review surface should let humans move from overview to exact evidence fast, because GitHub’s existing PR review model already centers commits, changed files, file filters, comments, and discussions, while CODEOWNERS centers responsibility. ## Evaluation, priorities, roadmap, and open questions

Most current benchmarks for coding agents measure whether a patch solved a task, not whether a human could safely understand, verify, compare, or roll it back. SWE-bench Verified is a useful patch-correctness signal because it is a human-validated subset of real GitHub issue tasks, but it is not a debuggability benchmark; it does not measure time to identify regression cause, explanation accuracy, or trust calibration. Teams building multi-agent coding systems should therefore add explicit product metrics for debuggability rather than treating patch success as sufficient. citeturn11search3turn11search10turn11search12

The right evaluation framework combines outcome metrics, process metrics, and trust metrics. Outcome metrics should include time to understand a change, time to identify regression cause, time to approve a PR safely, rollback speed, hidden regression catch rate, unsupported-claim catch rate, failed-merge reduction, flaky-test reduction, coverage of changed behaviors, and attribution accuracy. Process metrics should include duplicate-work reduction, handoff completeness, branch/worktree conflict frequency, evidence completeness, and percentage of changes with passing visual plus functional plus accessibility evidence. Trust metrics should include calibrated trust, false confidence rate, user-reported cognitive load, and reviewer satisfaction. These metrics should be tested on benchmark tasks such as: debugging a subtle UI regression with a tiny code diff, attributing a failing Playwright trace to one of three competing agent patches, reviewing a multi-agent auth refactor with changed dependencies and migrations, and selectively rolling back one agent’s contribution without breaking another’s dependent fix. Those tasks are closer to real platform needs than pure issue-resolution leaderboards.

The highest-priority recommendations are as follows.

**Must-have.** A unified event model with correlation IDs; per-step agent state snapshots; provenance ledger entries for file reads, writes, commands, browser actions, and tests; Playwright trace and screenshot ingestion; semantic intent grouping for diffs; PR evidence cards with unsupported-claim warnings; branch/worktree isolation; checkpoints or savepoints; and risk scoring based on blast radius, critical paths, and test evidence. These are the minimum viable foundation because they directly answer “who changed what, why, and with what evidence?”

**Should-have.** Browser provenance overlays; component-to-source attribution; collaboration timeline and handoff cards; context and memory inspector; conflict maps; UI-region coverage overlays; accessibility/security/privacy diff panels; and rollback-by-task with causal previews. These materially reduce review and debugging time, but depend on the must-have instrumentation layer.

**Advanced.** Alternative-solution comparison, semantic partial merges, runtime-effect attribution, causal rollback of dependency or migration changes, trajectory-quality scoring, deployment-readiness reports, and signed provenance attestations. These features deepen governance and operations, especially for enterprise or regulated environments.

**Experimental.** Live component-level heatmaps of agent effort, automatic hidden-consequence labeling, semantic user-journey coverage projection, explanation-quality scoring with human feedback loops, and CRDT-backed collaborative agent editing. These can be valuable, but they invite false precision unless the underlying signals are already strong.

A practical roadmap should look like this.

**First 30 days.** Instrument the agent runtime and CI pipeline. Emit stable request, task, step, and artefact IDs. Capture file reads/writes, commands, tests, and prompts used. Store Playwright traces, screenshots, structured logs, and Git patches. Build a basic PR evidence drawer showing agent summary, changed files, related tests, and raw artefact links. Add unsupported-claim warnings when summaries reference no artefacts.

**First 90 days.** Add semantic diff clustering, provenance search, risk scoring, checkpoints, worktree-per-agent orchestration, and an agent timeline with handoffs. Integrate source-map and framework-devtools mappings for component attribution. Ship visual diff cards, accessibility evidence cards, and test evidence cards. Add rollback previews for task clusters.

**First 6 months.** Build the browser provenance overlay and hidden inspection layer. Add UI-region coverage projection from Playwright traces. Introduce conflict maps, memory/context inspection, and causal graphs. Move the telemetry model onto OpenTelemetry conventions and enrich spans with coding-agent semantics. Introduce policy-driven approval gates for high-risk changes.

**First 12 months.** Add signed provenance export, advanced replay and selective rollback, side-by-side alternative solution comparison, runtime-effect attribution into production observability, deployment-readiness reporting, and organization-level analytics for explanation accuracy, reviewer latency, and hidden regression capture. At this stage the product ceases to be “an AI editor with logs” and becomes a governed software-change operating system.

**Open questions / limitations.** The biggest unresolved questions are standardization and storage. There is not yet a common cross-vendor schema for coding-agent provenance comparable to OpenTelemetry or OpenLineage. Cross-framework component attribution is still uneven. Deterministic replay of agent systems remains partial because LLM and external tool calls are inherently non-deterministic unless recorded and cached. Rich evidence capture can also create serious privacy, security, and storage burdens, especially for screenshots, browser traces, and memory snapshots. The clearest final recommendation is this: **treat every AI-made code change as a distributed systems event, a review artifact, and a user-experience change all at once**. If teams build around that premise, then multi-agent coding can become governable and trustworthy. If they do not, they will keep shipping systems that can write code faster than humans can understand it.
