# Cell-Based Modularization and Splitting Framework for Autonomous Development Systems

## Overview

Modern guidance on monolith vs. microservices converges on the idea of a spectrum: start with a modular monolith, then split only when there is strong evidence that different parts need different operational characteristics such as scaling, reliability, or deployment cadence. Cohesion and coupling remain the core technical criteria: healthy modules have high internal cohesion and low external coupling, making them easier to change, test, and eventually extract if needed. This report reframes those ideas in terms of "cells" in a living organism and adds concrete decision rules and scoring that an autonomous AI development system can apply.[^1][^2][^3][^4][^5][^6][^7][^8][^9]

Because there is no specific repository or runtime attached to this query, all references to "the system" are generic and must be instantiated against an actual codebase via metrics collection (change history, build logs, test times, dependency graphs) and architecture discovery tools. Where evidence is required (e.g., change frequency or failure patterns), the report specifies what must be measured rather than assuming values.[^10][^6]

## 1. Current natural shape of the system

In the absence of the concrete codebase, the "natural shape" of a system is determined by:

- Runtime behaviours: main user journeys, background jobs, batch processes, and external event processing.
- Data flows: how data enters (APIs, UIs, queues), where it is persisted, transformed, and delivered.
- State ownership: which components own authoritative data vs. read-only projections or caches.
- Failure modes: which failures are local and which cascade across the system.

For a real system, an autonomous agent should collect the following evidence:

- Static dependency graph from code (imports, module references, service calls) to identify clusters.
- Runtime traces or logs that show typical call graphs and long critical paths.
- Git history linking files that change together and the reasons for change (commit messages, PR labels).
- Test suite boundaries: which tests run where, how long they take, which components they touch.
- Deployment topology: what is deployed together, which processes/services exist, and their runtime configs.

These data allow clustering (e.g., via graph algorithms) to reveal de facto modules: groups of files and runtime components that frequently interact and change together. This is the "natural shape" rather than the intended architecture.[^5][^1]

## 2. Where the system is already trying to become modular

Empirical signals that a system is trying to modularise include:

- Repeated concepts: similar types, DTOs, or domain objects appearing in different packages or services.
- Duplicated logic: copied validation, pricing rules, permission checks, or parsing logic.
- Overloaded files or classes: god classes that reference many unrelated concepts.
- Unstable areas: directories whose files change in most releases, especially with many distinct reasons.
- Tests that require long setup: e.g., end-to-end tests needed to validate small logic changes.
- Shared state misuse: many modules writing to the same global or database tables without clear ownership.
- Tangled dependencies: cycles in module imports or services calling each other in loops.

An automated system can detect these with:

- Static analysis (graph cycles, size and complexity metrics, dependency fan-in/fan-out).
- Git mining (change frequency, co-change clusters, files touched per PR).
- Test telemetry (which tests run for which files, test duration, flakiness rate).

These areas are candidates for "watch zones": emerging boundaries under pressure but not yet stable.

## 3. Real split candidates and cell definition

A "cell" is a module with:

- Single clear responsibility: a cohesive domain capability or technical function.[^2][^5]
- Stable input–output contract: explicit APIs, messages, or function signatures.
- Clear state boundary: it owns some data or delegates ownership through a stable interface.
- Testable behaviour: local tests can validate its core responsibilities without full-system spin-up.
- Low shared mutable state: minimal need to share writable data structures.
- High internal cohesion: elements inside mostly relate to the same concept or workflow.
- Manageable external dependencies: few, well-understood dependencies.
- Independent evolution pressure: changes often localised to this responsibility.
- Measurable feedback improvement: splitting or internal extraction shortens build/test/debug cycles.

Cells do not have to be separate services; they can exist as internal modules, packages, or libraries within a single process. This aligns with the "modular monolith" pattern, where strong internal modules are the precursor to any future microservice or agent separation.[^4][^8][^1]

## 4. When the system should split: decision framework

An evidence-based decision framework can be expressed as a state machine over modules plus quantitative triggers.

### Qualitative triggers

A split from a larger module or cell boundary should be considered when:

- Multiple reasons to change: commits for the module consistently involve unrelated concerns (e.g., user billing and email templates in the same module).
- Comprehension limits: the module exceeds what a single agent (or human) can fully load into working memory, measured via file count, cyclomatic complexity, and dependency fan-out.[^3][^2]
- Test isolation failure: testing a small feature requires running slow integration or end-to-end suites.
- Development waits: significant wall-clock time is spent waiting on shared resources (builds, environments, long tests).
- Cross-area breakages: changes in one feature area repeatedly break others, indicating hidden coupling.
- Divergent change rhythms: some parts change weekly, others quarterly, but share the same deployment and test boundary.
- Different operational needs: parts require different scaling, reliability, security, or deployment cadence.
- Coordination cost exceeds interface cost: time spent aligning developers/agents within the module exceeds an estimate of defining and maintaining an explicit contract.
- Stable pattern: the same pattern or capability has been reimplemented or cloned at least three times.
- Simple contract: the boundary can be expressed as a small, clear interface description.
- Feedback-time reduction: measured or predicted test/build time reductions after split.

### Quantitative thresholds (example, to be calibrated per system)

An autonomous system can use heuristics such as:

- Average PR touches more than N files in this module that belong to distinct responsibilities.
- Module cyclomatic complexity or LOC exceeds configurable thresholds.
- Median test time for changes in this module exceeds a threshold compared to the rest of the system.
- Defect rate in this module is significantly higher than baseline.

These thresholds should be derived empirically from the system’s history rather than hard-coded; for example, by looking at distributions and flagging outliers.

## 5. When the system should not split

Premature or speculative splitting often causes more complexity than it removes, a problem widely observed in microservice overuse. A split should be actively avoided when:[^8][^1][^10][^4]

- Domain uncertainty: the domain model or requirements are still fluid and boundaries are likely to change soon.
- Speculative interface: the contract would be mostly guessed rather than anchored in existing usage patterns.
- Heavy shared mutable state: the potential child module would constantly need read-write access to parent internals.
- Over-fragmentation risk: split would create many tiny modules with high coordination overhead.
- Coupling concealment: the split would hide strong coupling behind network or interface layers without reducing it.
- Test cost increase: tests for the feature would become slower or more complex (e.g., requiring network hops).
- No independent lifecycle: the capability never changes or deploys independently of its parent.
- Insufficient pressure: the boundary has not experienced repeated issues (few co-changes, low wait time).

Autonomous agents must treat these as hard brakes: if most of these conditions hold, only refactor internally (rename, reorganise, extract pure functions) rather than splitting cells or services.

## 6. Designing from the beginning for easy future splitting

A system designed as a modular monolith from the start can be split later with far less risk and cost.[^1][^4][^8]

Key initial design rules:

- Responsibility-first modules: group code by business capability or cohesive technical responsibility, not only by layer (e.g., avoid massive "services" or "controllers" packages mixing many domains).[^5][^1]
- Explicit data ownership: each aggregate or table has a clear owning module; others access it via interfaces or queries that keep write ownership central.[^4]
- Isolated side effects: confine I/O, network, and external APIs behind well-defined ports/adapters.
- Small, stable interfaces: minimise the number of functions and DTO fields exposed across module boundaries.
- Orchestration vs. execution: keep workflows/orchestrators separate from domain logic so orchestration can move without rewriting core rules.
- Transport/storage independence: domain code should not depend directly on HTTP, gRPC, SQL drivers, or UI libraries.
- Local testability: ensure each module can be tested via its public contract with fast unit and integration tests.
- Future cell boundary thinking: design modules as if they could be extracted later, but keep them in-process until evidence emerges.
- Early metric tracking: instrument build times, test times, failure rates, and co-change metrics from the first releases.
- Controlled global state: any global or shared state is encapsulated in a dedicated owner module.

These rules align with domain-driven design (bounded contexts) and modular monolith practices that are now common recommendations.[^1][^4][^5]

## 7. Agent decision protocol for specialisation

An autonomous development agent implementing a change should execute an explicit protocol before deciding to:

- Modify existing code in-place.
- Extract an internal cell.
- Propose a larger architectural split.

### Pre-change questions

For each proposed change, the agent answers (using code analysis and history):

1. Does this change belong to an existing responsibility? If yes, identify the owning module or cell.
2. Does it introduce a new responsibility? If so, is it closely related to an existing cell or concept?
3. Does it increase coupling, e.g., by adding new dependencies or cross-module calls?
4. Does it introduce a new data owner, table, or aggregate?
5. Does it require a distinct test strategy (e.g., heavy load testing, security tests, offline simulations)?
6. Does it make feedback slower (more tests, longer build paths, or additional environments)?
7. Does it significantly increase module complexity (size, number of concepts)?
8. Is this capability: first occurrence, second similar occurrence, or third repeated pattern?
9. Would a local module contract make the change safer and easier to test?
10. Could this capability be tested independently if extracted?

### Protocol rules

- On first occurrence: implement inside the existing module, but annotate as a potential "watch zone" if it seems like a new responsibility.
- On second similar occurrence: refactor the common behaviour into an internal abstraction within the same module (e.g., helper or sub-package), strengthen tests, and mark the abstraction as a watch zone.
- On third repeated pattern with stable usage: evaluate for extraction into a cell according to the scoring model (Section 8).
- Only consider splitting into a separate package, service, or agent role when evidence about independent lifecycle, security isolation, runtime isolation, or testing isolation is strong (e.g., different SLAs, regulatory constraints).

## 8. Measurable signals and scoring model

To operationalise decisions, define a 0–5 score for each factor for a candidate module/cell:

- 0: Not present / no evidence.
- 1–2: Weak/moderate.
- 3–4: Strong.
- 5: Extreme.

### Factors (0–5 each)

1. **Cohesion**: How focused is the module on a single responsibility?
2. **Coupling**: How strongly does it depend on (or is depended on by) the rest of the system?
3. **Change frequency**: How often does it change compared to others?
4. **Independent testability**: How easily can it be tested in isolation?
5. **Feedback-time reduction**: Expected reduction in build/test/debug time if split.
6. **Interface stability**: Likelihood that a public contract would remain stable.
7. **State ownership clarity**: How clear are data ownership boundaries?
8. **Failure isolation benefit**: How much would splits confine failures?
9. **Cognitive-load reduction**: How much would splits reduce per-change mental load?
10. **Deployment/runtime independence**: Does it need different scaling, deployment, or runtime settings?
11. **Risk of premature abstraction**: Risk that the boundary is speculative.
12. **Migration difficulty**: How hard would it be to move existing code and dependencies?

### Example scoring interpretation

- High cohesion (4–5) and high coupling (4–5) suggests the module is a good candidate for internal extraction but must be decoupled before service-level splits.
- High change frequency (4–5), high independent testability potential (3–5), and strong feedback-time reduction (3–5) push toward extraction.
- High risk of premature abstraction (4–5) or high migration difficulty (4–5) argues for "watch zone" rather than immediate split.

### Decision categories

Based on factor scores and qualitative triggers:

- **Keep unified**: Overall need low; risk and cost of splitting outweigh benefits; cohesion may be moderate, but coupling manageable, and feedback-time gains small.
- **Mark as watch zone**: Some pressure (change frequency, budding new responsibility) but interface and ownership not yet stable.
- **Extract internal module**: Strong evidence for new responsibility and testability improvement, but no need for separate deployment or runtime.
- **Extract package/library**: Capability useful across multiple modules/services; stable APIs; high cohesion and reusability.
- **Extract service**: Clear need for independent deployment, scaling, or security; strong isolation benefits.
- **Extract agent role**: Distinct behavioural policy or reasoning function with its own prompts, constraints, or tools (especially in AI systems).[^11][^12]
- **Extract plugin**: Optional or user-extensible capability with well-defined extension points.
- **Merge back**: Prior split did not deliver expected benefits; coupling and coordination remain high.

The decision engine should emit not only the decision but also factor scores, metrics used, and links to evidence (commits, test runs, traces) so future agents can audit and adjust thresholds.

## 9. Cell lifecycle

A cell (emerging module) progresses through stages:

1. **Seed**: New behaviour appears once inside a parent module. Implementation is local, and responsibility is tentative.
2. **Watch zone**: Behaviour appears again or causes friction (bugs, slow tests, confusion). The boundary is suspected but not yet formalised.
3. **Budding cell**: The behaviour is refactored into an internal abstraction (sub-package, class, or component) with a draft contract and local tests.
4. **Independent cell**: The abstraction has a stable responsibility, clear interface, ownership, and tests. Other modules use it as a dependency.
5. **Specialised cell**: The cell has its own operational characteristics: separate build/test pipeline, security profile, deployment cadence, or dedicated agent ownership.
6. **Diseased cell**: The cell accumulates issues: high coupling, unclear ownership, duplication, slow tests, or frequent incidents.
7. **Merge or repair**: The cell is simplified, merged back into parent(s), or given a cleaned-up boundary.

Agents should maintain metadata about each cell (stage, owner, metrics) in a registry so lifecycle decisions are explicit and trackable.

## 10. Refactoring rules for safe splits

When a split is justified, safe refactoring principles from monolith-to-microservice experience apply.[^8][^4][^1]

- Preserve behaviour first: avoid new features during split.
- Add characterisation tests: capture current behaviour at the intended boundary.
- Define the public contract before extraction: inputs, outputs, errors.
- Move one responsibility at a time: avoid big-bang moves.
- Use strangler or adapter patterns: keep old and new paths in parallel during migration when necessary.
- Clarify state ownership: avoid shared mutable state; adopt patterns like transaction scripts, event sourcing, or sagas when necessary for consistency.
- Add local fast tests: unit and small integration tests for the new cell.
- Add contract tests: ensure interactions between modules match the contract.
- Reserve end-to-end tests for critical cross-cell flows.
- Measure feedback times: compare build and test durations before and after.
- Remove dead code: delete old paths and obsolete abstractions once stable.
- Update documentation and agent instructions: so future changes respect the new boundary.
- Define rollback criteria: metrics and incident thresholds that trigger revert or merge-back.

## 11. Independent development readiness

A module is ready for independent development (separate agent, team, or service) when:

- Contract explicitness: interface fully documented, versioned, and discoverable.
- Stable inputs/outputs: changes to the contract are infrequent and carefully managed.
- Local testability: full test suite can run without spinning up the entire system (using mocks, stubs, or local dependencies).
- Dependency injection: external dependencies are injectable and mockable.
- Clear data access: owns its data or accesses external data through stable APIs.
- Known failure modes: documented error conditions and their handling.
- Clear responsibility: succinct description of what it does and does not do.
- Local changeability: internal changes rarely require changes in unrelated modules.
- Ownership: an agent or team is responsible for its health and evolution.[^10]
- Independent acceptance criteria: success metrics and test oracles defined for the module itself.
- Contract tests: automated checks against consumers and providers of its API.

If several of these are missing, the system should invest in internal modularisation and test design before splitting services or agents.

## 12. Feedback-time strategy

Fast feedback is one of the strongest practical reasons for modularisation. The testing strategy should reflect cell boundaries:[^4][^8][^1]

- Unit tests: focus on internal behaviour of each cell, fast and run on every small change.
- Contract tests: verify that cell interfaces satisfy consumer expectations.
- Integration tests: validate collaborations of a small number of cells for critical scenarios.
- End-to-end tests: verify core user journeys; small in number due to cost.
- Selective test execution: test runner maps code changes to affected cells and runs only relevant tests by default.
- Risk-based gating: slow, system-wide tests triggered for high-risk changes or before releases, not for every commit.

Every new cell should reduce unnecessary waiting time. If a proposed split would increase or maintain the same feedback time, it should be reconsidered.

## 13. Architecture outputs for an autonomous system

For a concrete system, an autonomous architecture agent should be able to generate and maintain:

- **Current system map**: processes, modules, services, agents, and their interactions (call graphs, data flows).
- **Responsibility map**: mapping responsibilities/capabilities to modules/cells.
- **Dependency map**: static and runtime dependencies.
- **Data ownership map**: which module owns which tables, aggregates, or external resources.
- **Test boundary map**: which tests exercise which modules, their duration, and flakiness.
- **Feedback-time bottlenecks**: slowest tests, builds, and long critical paths.
- **Watch zones**: modules or areas under observation due to potential new boundaries.
- **Split candidates**: modules scoring high on change frequency, coupling, and feedback-time benefits.
- **No-split candidates**: modules with low pressure and high risk of premature abstraction.
- **Recommended first refactor**: the smallest high-leverage refactor that improves feedback without large risk.
- **Recommended future refactors**: potential future splits conditional on observed metrics.
- **Risks of splitting**: expected migration difficulties, potential downtime, or temporary complexity.
- **Risks of not splitting**: continued slow feedback, higher error rates, coordination cost.
- **Scoring table**: per-module/cell factor scores from Section 8.
- **Migration plan**: step-by-step refactor plan for selected splits.
- **Test plan**: tests to add or adjust pre- and post-split.
- **Contract definitions**: explicit interface specs for split modules.
- **Agent rules**: instructions for agents making future changes on how to respect boundaries.
- **Monitoring signals**: metrics showing whether splits improved feedback, reliability, and coupling.

## 14. Success criteria for a split

A split is considered successful if, after a bedding-in period:

- Responsibility clarity: the new module has a more precise, simpler responsibility description.
- Test improvements: tests for that responsibility are faster, more focused, or easier to write and run.
- Reduced co-change: fewer unrelated files and modules change together with that module.
- Smaller interface: the explicit contract surface is smaller than the previous implicit coupling.
- Cognitive ease: developers/agents report or metrics indicate faster understanding and change implementation.
- Reduced wait time: build and test pipelines show shorter run times for changes confined to the cell.
- Easier defect localisation: incidents can be traced quickly to a specific cell.
- Independent evolution: module can be versioned or deployed without frequent coordination.
- No net complexity increase: the additional infrastructure or interfaces are justified by these benefits.

These can be quantified with before/after metrics on test times, change sizes, incident MTTR, and co-change graphs.

## 15. Failure criteria for a split

A split is considered to have failed (fully or partially) if:

- Frequent cross-changes: most changes to the new module require coordinated PRs in the parent or consumers.
- Unstable interface: the module’s contract changes often and in breaking ways.
- Slower tests: relevant test suites are slower or more fragile than before.
- Unclear data ownership: multiple modules write to the same data without clear coordination.
- Ownership confusion: agents or teams regularly mis-route changes or bug fixes.
- Over-splitting: the module is too small or trivial to justify its boundary.
- Logic duplication: similar logic now exists in multiple modules due to split.
- Extra integration overhead: more cross-module coordination is needed for the same feature.
- Coupling unchanged: underlying coupling remains high despite new boundaries.

When these signals persist, the system should consider merging back or redefining the cell boundary.

## 16. Final recommendations for autonomous systems

For an autonomous development system with no human in the loop:

- Maintain a live architecture registry (modules, cells, responsibilities, scores, lifecycle stage).
- On each change, run the agent decision protocol and update factor scores for affected modules.
- Treat splits as experiments with hypotheses about feedback-time reduction and coupling improvement.
- Instrument and compare metrics before and after each split.
- Apply the cell lifecycle stages to decide when to promote a watch zone to budding cell or independent cell.
- Be conservative: prefer internal extraction and refactoring before any network or process boundary.
- Regularly re-evaluate diseased cells and failed splits for merge or boundary repair.

This framework aims to make specialisation in software systems more like biological differentiation: gradual, evidence-driven, reversible when necessary, and always anchored in the health of the overall organism.[^12][^13][^14][^11]
